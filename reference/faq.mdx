---
title: "SNI-Spoofing FAQ: platforms, SNI behavior, and building EXEs"
description: "Answers to common questions about SNI-Spoofing: supported platforms, how the fake SNI works, security considerations, and performance."
---

SNI-Spoofing works by intercepting the TCP handshake at the kernel level and injecting a crafted TLS ClientHello that contains a decoy hostname. The questions below cover the most common points of confusion about platform support, how the technique works in practice, and how to build and distribute the tool.

<AccordionGroup>

  <Accordion title="Does this work on Linux or macOS?">
    No. SNI-Spoofing depends on [WinDivert](https://reqrypt.org/windivert.html), a Windows kernel driver that captures and re-injects network packets. WinDivert has no Linux or macOS equivalent, and neither does the pydivert Python binding. The tool must run on a Windows host or a Windows virtual machine.
  </Accordion>

  <Accordion title="What does FAKE_SNI do exactly?">
    `FAKE_SNI` is the hostname embedded in the fake TLS ClientHello that `fake_send_thread` injects. The injected packet is sent with a deliberately wrong TCP sequence number — specifically `syn_seq + 1 - len(fake_data)` — which places it below the server's current receive window. The real server therefore discards it as an out-of-window segment. However, DPI systems that inspect all passing TCP segments (not only in-window ones) read the ClientHello and see `FAKE_SNI` as the requested hostname. The actual TLS handshake then proceeds normally with the real ClientHello sent by your application.
  </Accordion>

  <Accordion title="Does the FAKE_SNI domain need to be reachable?">
    No. The server at `CONNECT_IP` discards the fake ClientHello because its sequence number is out of window. No connection is ever established to the `FAKE_SNI` domain; the hostname only needs to be one that your network's DPI policy treats as allowed.
  </Accordion>

  <Accordion title="Can I use this with any protocol?">
    Currently only TLS is supported. The `DATA_MODE` variable in `main.py` is hardcoded to `"tls"`, and `handle()` exits with an error for any other value. The fake packet is always a TLS 1.3 ClientHello built by `ClientHelloMaker.get_client_hello_with()`. HTTP and other protocol modes are not implemented.
  </Accordion>

  <Accordion title="What is WinDivert and why does it need admin rights?">
    WinDivert is an open-source Windows kernel driver that lets user-space programs intercept, inspect, and re-inject network packets before they are processed by the OS network stack. Because it operates at the kernel level, loading the driver requires administrator privileges. pydivert installs and manages the WinDivert driver automatically; you do not need to install the driver separately. You do, however, need to run `python main.py` (or the compiled EXE) as an administrator every time.
  </Accordion>

  <Accordion title="Will this work against all DPI systems?">
    No. SNI-Spoofing works against DPI systems that inspect out-of-window TCP segments — a technique sometimes called "ghost injection" or "out-of-order injection." DPI implementations that track TCP receive windows and discard out-of-window data will not see the fake SNI. Effectiveness depends entirely on the specific DPI hardware and software deployed by your network. Results vary significantly between networks and ISPs.
  </Accordion>

  <Accordion title="How do I build a single-file EXE?">
    Use PyInstaller with the `--onefile` flag:

    ```bash
    pip install pyinstaller
    pyinstaller --onefile main.py
    ```

    The compiled EXE is placed in the `dist/` directory. `get_exe_dir()` in `main.py` detects the PyInstaller frozen environment via `sys.frozen` and resolves `config.json` relative to the EXE rather than `__file__`, so the binary works correctly from any working directory. Place `config.json` in the same directory as the EXE before running it.
  </Accordion>

</AccordionGroup>
